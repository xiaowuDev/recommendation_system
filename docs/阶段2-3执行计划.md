## 二、阶段 2 — 数据接入执行（后端）

### 2.1 目标

把阶段 1 的"连接可配置、可测试"升级为"连接可配置、可测试、可导入"。用户可以针对一条已保存的连接，手动触发一次数据导入任务，系统拉取/解析数据后将结果记录到 `etl_ingestion_job`。

### 2.2 领域模型设计

#### 2.2.1 IngestionJob 实体

```
datasource/domain/entity/IngestionJob.java
```

| 字段 | 类型 | 说明 |
|------|------|------|
| id | Long | 主键，自增 |
| connectionId | Long | 关联 DataSourceConnection |
| dataSourceType | DataSourceType | 数据源类型 |
| status | IngestionJobStatus | PENDING → RUNNING → COMPLETED / FAILED |
| message | String | 状态描述或错误信息 |
| request | IngestionRequest | 请求参数（序列化为 JSON 存储） |
| result | IngestionResult | 执行结果摘要（序列化为 JSON 存储） |
| startedAt | LocalDateTime | 任务开始时间 |
| finishedAt | LocalDateTime | 任务结束时间（nullable） |

#### 2.2.2 IngestionJobStatus 枚举

```java
public enum IngestionJobStatus {
    PENDING,    // 已创建，等待执行
    RUNNING,    // 执行中
    COMPLETED,  // 执行成功
    FAILED      // 执行失败
}
```

#### 2.2.3 IngestionRequest（sealed interface）

```
datasource/domain/model/IngestionRequest.java
```

按数据源类型分四种实现，统一走 sealed interface：

| 类型 | 实现类 | 核心参数 |
|------|--------|----------|
| KAFKA | KafkaIngestionRequest | topic, maxRecords(默认100), timeoutMs(默认10000) |
| MYSQL | MysqlIngestionRequest | query(SQL语句), maxRows(默认1000) |
| REDIS | RedisIngestionRequest | keyPattern, scanCount(默认100), maxKeys(默认100) |
| EXCEL | ExcelIngestionRequest | sheetName, headerRowIndex, maxRows(默认1000) |

#### 2.2.4 IngestionResult（record）

```
datasource/domain/model/IngestionResult.java
```

| 字段 | 类型 | 说明 |
|------|------|------|
| totalRecords | int | 读取到的总记录数 |
| columns | List<String> | 列名/字段名列表 |
| sampleRows | List<Map<String, Object>> | 前 N 行采样数据（默认前 20 行） |
| metadata | Map<String, Object> | 额外元数据（如 Kafka offset、MySQL 查询耗时等） |

### 2.4 数据接入执行器（Port + Adapter 模式）

#### 2.4.1 DataIngester 接口（应用层端口）

```
datasource/application/port/DataIngester.java
```

```java
public interface DataIngester {
    DataSourceType supports();
    IngestionResult ingest(DataSourceConfig config, IngestionRequest request);
}
```

与 `ConnectionTester` 模式完全平行，方便 Spring 自动收集和按类型分发。

#### 2.4.2 四类实现

| 实现类 | 位置 | 核心逻辑 |
|--------|------|----------|
| KafkaDataIngester | `infrastructure/ingestion/KafkaDataIngester.java` | 创建临时 KafkaConsumer，assign 到指定 topic 的所有分区，poll maxRecords 条消息，解析为 Map，关闭消费者 |
| MysqlDataIngester | `infrastructure/ingestion/MysqlDataIngester.java` | 用独立 JDBC 连接执行用户 SQL（只读，设 readOnly=true），ResultSet 映射为 List<Map>，限制 maxRows |
| RedisDataIngester | `infrastructure/ingestion/RedisDataIngester.java` | 用 SCAN 遍历 keyPattern，对每个 key 做 TYPE + 对应读取（STRING→GET, HASH→HGETALL, LIST→LRANGE, SET→SMEMBERS, ZSET→ZRANGEWITHSCORES），限制 maxKeys |
| ExcelDataIngester | `infrastructure/ingestion/ExcelDataIngester.java` | 复用阶段 1 的 POI 逻辑，但读取完整数据（不再只是 sampleSize 行），返回标准化 IngestionResult |

#### 2.4.3 安全约束

- **MySQL**: 只允许 SELECT 语句，service 层验证 SQL 不含 INSERT/UPDATE/DELETE/DROP/ALTER/TRUNCATE（正则白名单）
- **Redis**: 使用 SCAN 而非 KEYS，强制限制 maxKeys 上限为 10000
- **Kafka**: 消费者设置 `max.poll.records` 和超时，避免无限阻塞
- **Excel**: 复用上传文件，不走网络连接；maxRows 上限 50000

### 2.5 应用层服务

#### 2.5.1 IngestionJobApplicationService

```
datasource/application/service/IngestionJobApplicationService.java
```

核心方法：

```java
/**
 * 触发一次导入任务（同步执行）
 * 1. 校验 connection 存在
 * 2. 创建 PENDING job 并持久化
 * 3. 更新为 RUNNING
 * 4. 分发到对应 DataIngester 执行
 * 5. 成功 → COMPLETED + 存储结果；失败 → FAILED + 存储错误
 * 6. 返回完整 IngestionJob
 */
IngestionJob execute(Long connectionId, JsonNode requestJson);

/**
 * Excel 导入（需要 MultipartFile）
 * 流程同上，但从文件解析而非网络连接
 */
IngestionJob executeExcel(MultipartFile file, JsonNode requestJson);

/** 查询单个 job 详情 */
IngestionJob getJob(Long jobId);

/** 查询某连接最近的 job 列表 */
List<IngestionJob> recentJobs(Long connectionId, int limit);
```

```
datasource/application/support/IngestionRequestMapper.java
```

### 2.9 验收标准

1. 已保存的 Kafka 连接 → 触发导入 → 返回消息列表（或超时无数据时返回空 + 提示）
2. 已保存的 MySQL 连接 → 触发 `SELECT * FROM xxx LIMIT 10` → 返回查询结果
3. 已保存的 Redis 连接 → 触发 `supplier:*` scan → 返回 key-value 列表
4. 上传 Excel 文件 → 触发导入 → 返回完整数据（受 maxRows 限制）
5. 非 SELECT SQL → 400 拒绝
6. job 状态可查询，失败 job 有错误信息

### 3.4 Pretext 集成方案

Pretext 是一个轻量级标记语言，项目根目录已有 `pretext-0.3.0.tgz`。在前端中的使用方式：

#### 3.4.1 安装

```bash
cd datasource-ui
npm install ../pretext-0.3.0.tgz
```

#### 3.4.2 PretextRenderer 组件

```vue
<!-- components/shared/PretextRenderer.vue -->
<script setup lang="ts">
import { computed } from 'vue'
import pretext from 'pretext'

const props = defineProps<{ source: string }>()
const rendered = computed(() => pretext(props.source))
</script>

<template>
  <div class="pretext-prose" v-html="rendered" />
</template>
```

#### 3.4.3 usePretext composable

```ts
// composables/usePretext.ts
import pretext from 'pretext'
import { computed, type Ref } from 'vue'

export function usePretext(source: Ref<string>) {
  const html = computed(() => pretext(source.value))
  return { html }
}
```

#### 3.4.4 使用场景

- **连接描述**：连接的 `description` 字段可以使用 Pretext 语法编写，前端渲染为富文本
- **导入任务消息**：`IngestionJob.message` 中的文本使用 Pretext 渲染
- **帮助文档/提示**：页面内嵌的说明文本使用 Pretext 格式
- **测试结果展示**：连接测试结果的 message 字段使用 Pretext 渲染

### 3.7 API 调用封装

```ts
// api/client.ts
const BASE_URL = '/api/examples/data-sources'

async function request<T>(path: string, options?: RequestInit): Promise<T> {
  const response = await fetch(`${BASE_URL}${path}`, {
    headers: { 'Content-Type': 'application/json', ...options?.headers },
    ...options,
  })
  if (!response.ok) {
    const error = await response.json().catch(() => ({ message: response.statusText }))
    throw new ApiError(response.status, error.message)
  }
  return response.json()
}
```

## 五、风险与对策

| 风险 | 对策 |
|------|------|
| MySQL 导入执行任意 SQL 有安全风险 | Service 层 SQL 白名单校验，只允许 SELECT；设置 JDBC readOnly + queryTimeout |
| Redis KEYS 命令阻塞生产 | 强制使用 SCAN，限制 maxKeys 上限 |
| Kafka 消费超时导致长时间阻塞 | 设置 poll timeout，整体方法加 timeout 控制 |
| Excel 大文件内存溢出 | 限制 maxRows，使用 SAX 模式解析（POI 的 SXSSF/EventModel 可选） |
| Pretext 0.3.0 是早期版本，可能有 bug | 对用户输入做 sanitize 再渲染，v-html 使用需注意 XSS |
| 前后端接口不一致 | 先完成后端 + Swagger UI 验证，再开始前端开发 |
